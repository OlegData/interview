# AWS

**What is cloud computing?**

---

**Cloud computing** is the delivery of computing services (servers, storage, databases, networking, software) over the internet on demand with pay-as-you-go pricing.

<br>

**Облачные вычисления** — это предоставление вычислительных ресурсов (серверы, хранилище, базы данных, сеть, программное обеспечение) через интернет по требованию с оплатой по мере использования.

**What is the problem with the traditional IT approach compared to using the Cloud?**

---

Traditional IT requires upfront capital investment in hardware, long procurement cycles, manual scaling, and ongoing maintenance, which reduces flexibility and increases costs.

<br>

Традиционный IT требует крупных первоначальных вложений в оборудование, длительных закупок, ручного масштабирования и постоянного обслуживания, что снижает гибкость и увеличивает затраты.

**How many types of deployment models exist in the cloud?**

---

There are four main cloud deployment models: Public, Private, Hybrid, and Community cloud.

Существует четыре основных модели развертывания облака: публичное, частное, гибридное и общественное облако.

**What are the five characteristics of cloud computing?**

---

**The five characteristics are**: On-demand self-service, Broad network access, Resource pooling, Rapid elasticity, and Measured service.
**On-demand self-service** — Users can provision computing resources automatically without human interaction with the provider.
**Broad network access** — Services are available over the network and accessed through standard devices like laptops or smartphones.
**Resource pooling** — Provider resources are shared among multiple customers using a multi-tenant model.
**Rapid elasticity** — Resources can be quickly scaled up or down based on demand.
**Measured service** — Resource usage is monitored, controlled, and billed based on consumption.

<br>

**Пять характеристик**: самообслуживание по требованию, широкий сетевой доступ, объединение ресурсов, быстрая эластичность и измеряемый сервис.
**Самообслуживание по требованию** — Пользователь может самостоятельно получать ресурсы без участия провайдера.
**Широкий сетевой доступ** — Сервисы доступны через сеть и стандартные устройства.
**Объединение ресурсов** — Ресурсы провайдера распределяются между несколькими клиентами.
**Быстрая эластичность** — Ресурсы быстро масштабируются в зависимости от нагрузки.
**Измеряемый сервис** — Использование ресурсов отслеживается и оплачивается по факту потребления.

**What are the main types of Cloud Computing?**

---

**The main types of Cloud Computing** are Infrastructure as a Service (IaaS), Platform as a Service (PaaS), and Software as a Service (SaaS).

Infrastructure as a Service (IaaS) provides virtualized computing resources such as servers, storage, and networking. The user manages the operating system and applications.

Platform as a Service (PaaS) provides a platform and environment for developing, testing, and deploying applications. The provider manages the infrastructure and operating system.

Software as a Service (SaaS) delivers ready-to-use applications over the internet. The provider manages everything.

<br>

**Основные типы облачных вычислений**: инфраструктура как услуга (IaaS), платформа как услуга (PaaS) и программное обеспечение как услуга (SaaS).

Инфраструктура как услуга (IaaS) предоставляет виртуальные серверы, хранилище и сеть. Пользователь управляет операционной системой и приложениями.

Платформа как услуга (PaaS) предоставляет среду для разработки, тестирования и развертывания приложений. Провайдер управляет инфраструктурой и операционной системой.

Программное обеспечение как услуга (SaaS) предоставляет готовые приложения через интернет. Провайдер управляет всем.

**What is Amazon EC2, and what are its main uses?**

---

**Amazon EC2 (Elastic Compute Cloud)** is a web service that provides scalable virtual servers in the cloud.

Its main uses include hosting applications, running websites, processing data, development and testing environments, and scaling workloads on demand.

<br>

**Amazon EC2** — это веб-сервис, предоставляющий масштабируемые виртуальные серверы в облаке.

Основные применения: хостинг приложений, запуск веб-сайтов, обработка данных, среды разработки и тестирования, масштабирование нагрузки по требованию.

**What is Amazon S3, and why is it important?**

---

**Amazon S3 (Simple Storage Service)** is an object storage service that stores and retrieves any amount of data over the internet.

It is important because it provides high durability, scalability, security, and cost-effective storage for backups, static websites, big data, and application data.

<br>

**Amazon S3** — это сервис объектного хранения данных, позволяющий хранить и получать любой объём данных через интернет.

Он важен благодаря высокой надёжности, масштабируемости, безопасности и экономичности для резервных копий, статических сайтов, больших данных и данных приложений.

**Explain the concept of \"Regions\" and \"Availability Zones\" in AWS**

---

**A Region** is a geographic area where AWS has multiple data centers.

An Availability Zone is an isolated data center within a Region, designed to provide high availability and fault tolerance.

<br>

**Регион** — это географическая область, где AWS размещает несколько дата-центров.

Зона доступности — это изолированный дата-центр внутри региона, предназначенный для высокой доступности и отказоустойчивости.

---

**What is IAM, and why is it important?**

**IAM (Identity and Access Management)** is a service that controls access to AWS resources by managing users, groups, roles, and permissions.

It is important because it ensures security by granting least-privilege access and protecting cloud resources.

<br>

**IAM** — это сервис управления доступом к ресурсам AWS через пользователей, группы, роли и разрешения.

Он важен для обеспечения безопасности и принципа минимально необходимого доступа.

**What is Amazon RDS, and how does it differ from traditional databases?**

---

**Amazon RDS (Relational Database Service)** is a managed service that runs relational databases in the cloud.

It differs from traditional databases because AWS handles provisioning, backups, patching, scaling, and maintenance, while the user focuses on the application and data.

<br>

**Amazon RDS** — это управляемый сервис реляционных баз данных в облаке.

Он отличается от традиционных БД тем, что AWS берет на себя развертывание, резервное копирование, обновления, масштабирование и обслуживание, а пользователь работает только с данными и приложением.

**What is Amazon VPC, and why is it used?**

---

**Amazon VPC (Virtual Private Cloud)** is a service that allows you to create a logically isolated virtual network in AWS.

It is used to control IP ranges, subnets, routing, internet access, and security for cloud resources.

<br>

**Amazon VPC** — это сервис, позволяющий создать логически изолированную виртуальную сеть в AWS.

Он используется для управления IP-адресами, подсетями, маршрутизацией, доступом в интернет и безопасностью облачных ресурсов.

**What is Amazon CloudWatch, and what are its main components?**

---

**Amazon CloudWatch** is a monitoring and observability service that collects, monitors, and analyzes metrics, logs, and events from AWS resources, applications, and on-premises systems. It helps track performance, detect issues, set automated alarms, and trigger actions.

Its main components are:
Metrics — numerical data about resource performance.
Logs — collection and analysis of log files.
Alarms — notifications or automated actions based on thresholds.
Dashboards — visual displays of metrics and logs.
Events (EventBridge) — event-based automation and response.

<br>

**Amazon CloudWatch** — это сервис мониторинга и наблюдаемости, который собирает, отслеживает и анализирует метрики, логи и события из ресурсов AWS, приложений и локальных систем. Он помогает контролировать производительность, выявлять проблемы, настраивать тревоги и запускать автоматические действия.

Основные компоненты:
Метрики — числовые показатели производительности.
Логи — сбор и анализ журналов.
Тревоги — уведомления или автоматические действия при превышении порогов.
Панели мониторинга — визуальное отображение метрик и логов.
События (EventBridge) — автоматизация и реакция на события.

**What is AWS Lambda, and how does it enable serverless computing?**

---

**AWS Lambda** is a serverless compute service that runs code in response to events without provisioning or managing servers. It enables serverless computing by automatically handling infrastructure, scaling, availability, and execution. You pay only for the compute time used.

<br>

**AWS Lambda** — это серверless-сервис вычислений, который запускает код в ответ на события без управления серверами. Он обеспечивает serverless-подход за счёт автоматического управления инфраструктурой, масштабирования и доступности. Оплата производится только за фактическое время выполнения кода.

**What is Elastic Load Balancing (ELB) in AWS?**

---

**Elastic Load Balancing (ELB)** is a service that automatically distributes incoming traffic across multiple targets such as EC2 instances, containers, or IP addresses.

It improves availability, fault tolerance, and scalability of applications.

<br>

**Elastic Load Balancing (ELB)** — это сервис, который автоматически распределяет входящий трафик между несколькими ресурсами, такими как EC2-инстансы, контейнеры или IP-адреса.

Он повышает доступность, отказоустойчивость и масштабируемость приложений.

**How do you use AWS CodePipeline to automate a CI/CD pipeline for a multi-tier application?**

---

Use CodePipeline to define stages that build, test, and deploy each tier automatically:

**Source**: connect a repo (CodeCommit/GitHub) and trigger on commit.

**Build/Test**: use CodeBuild to compile, run unit/integration tests, and package artifacts.

**Deploy**: deploy to each tier with CodeDeploy/Elastic Beanstalk/ECS/EKS/CloudFormation; use separate stages for dev → staging → prod.

**Artifacts**: store outputs in S3 and pass them between stages.

**Approvals/Quality gates**: add manual approval and automated checks before production.

**IAM/KMS**: use least-privilege roles and encrypt artifacts/secrets; use Parameter Store/Secrets Manager for configs.

<br>

В CodePipeline настраивают этапы, которые автоматически собирают, тестируют и развёртывают каждый слой приложения:

**Source**: подключить репозиторий (CodeCommit/GitHub) и запускать пайплайн по коммиту.

**Build/Test**: через CodeBuild собрать, прогнать unit/integration тесты и упаковать артефакты.

**Deploy**: развернуть по слоям через CodeDeploy/Elastic Beanstalk/ECS/EKS/CloudFormation; отдельные стадии для dev → staging → prod.

**Artifacts**: хранить артефакты в S3 и передавать между стадиями.

**Approvals/Quality gates**: добавить ручное подтверждение и автоматические проверки перед продом.

**IAM/KMS**: роли с минимальными правами, шифрование артефактов/секретов; конфиги через Parameter Store/Secrets Manager.

**What are some strategies for blue/green deployments on AWS?**
Common blue/green strategies on AWS:

ALB target groups — run blue and green behind one ALB and switch traffic by changing listener rules/weights.

Route 53 DNS switching — use weighted records or failover to shift traffic between environments.

AWS CodeDeploy blue/green — automate traffic shifting and rollback for EC2 and ECS.

Elastic Beanstalk — use environment swap to switch the CNAME between blue and green.

Lambda/API Gateway — use Lambda aliases with weighted routing to shift traffic gradually.

<br>

Основные стратегии blue/green на AWS:

ALB target groups — держать blue и green за одним ALB и переключать трафик правилами/весами.

Route 53 DNS — weighted/failover записи для перевода трафика между окружениями.

CodeDeploy blue/green — автоматизация переключения трафика и отката для EC2 и ECS.

Elastic Beanstalk — swap environments (смена CNAME) между blue и green.

Lambda/API Gateway — алиасы Lambda с взвешенной маршрутизацией для постепенного перевода трафика.

**How would you manage and secure secrets for a CI/CD pipeline in AWS?**

---

**Store secrets** in AWS Secrets Manager or SSM Parameter Store (SecureString) encrypted with KMS, and never hardcode them in code or build specs. Grant access via least-privilege IAM roles for CodeBuild/CodePipeline, rotate secrets, and audit access with CloudTrail. Inject secrets at runtime (env vars/parameters), limit who can read them, and use separate secrets per environment.

**Хранить секреты** в AWS Secrets Manager или SSM Parameter Store (SecureString) с шифрованием через KMS и никогда не зашивать их в код/скрипты сборки. Доступ давать через IAM роли с минимальными правами для CodeBuild/CodePipeline, включить ротацию, аудит доступа через CloudTrail. Подставлять секреты только во время выполнения (env vars/parameters), ограничить чтение и разделить секреты по окружениям.

**What is Auto Scaling?**

---

**Auto Scaling** is a service that automatically adjusts the number of compute resources based on demand and defined policies.

It improves availability, fault tolerance, and cost efficiency by scaling out during high load and scaling in during low load.

**Auto Scaling** — это сервис, который автоматически изменяет количество вычислительных ресурсов в зависимости от нагрузки и заданных политик.

Он повышает доступность, отказоустойчивость и экономичность, увеличивая ресурсы при высокой нагрузке и уменьшая при низкой.

**What is Amazon RDS?**

---

**Amazon RDS** (Relational Database Service) is a managed service that runs relational databases in the cloud, supporting engines like MySQL, PostgreSQL, MariaDB, Oracle, and SQL Server.

It automates provisioning, backups, patching, scaling, and high availability.

**Amazon RDS** — это управляемый сервис реляционных баз данных в облаке с поддержкой MySQL, PostgreSQL, MariaDB, Oracle и SQL Server.

Он автоматизирует развертывание, резервное копирование, обновления, масштабирование и высокую доступность.

---

It caches and distributes static and dynamic content such as websites, APIs, videos, and applications closer to users.

**Amazon CloudFront** — это сервис CDN, который доставляет контент с низкой задержкой и высокой скоростью через глобальную сеть edge-локаций.

Он кэширует и распределяет статический и динамический контент (сайты, API, видео, приложения) ближе к пользователям.

**What is the difference between a Security Group and a Network ACL?**

---

A **Security Group** is a stateful virtual firewall attached to an instance that controls inbound and outbound traffic at the instance level.

A **Network ACL** is a stateless firewall applied at the subnet level that controls inbound and outbound traffic using numbered rules.

**Security Group** — это stateful виртуальный firewall, привязанный к инстансу и управляющий входящим и исходящим трафиком на уровне инстанса.

**Network ACL** — это stateless firewall на уровне подсети, который управляет входящим и исходящим трафиком с помощью нумерованных правил.

**Amazon DynamoDB** — это полностью управляемая NoSQL база данных с высокой производительностью и автоматическим масштабированием.

Она поддерживает модели ключ-значение и документы, обеспечивая высокую доступность, надежность и низкую задержку.

**What is Amazon SNS and its use case?**

---

**Amazon SNS (Simple Notification Service)** is a fully managed pub/sub messaging service that enables message delivery to multiple subscribers.

**What is the difference between stopping and terminating an EC2 instance?**

---

Stopping an EC2 instance shuts it down but preserves the instance and attached EBS volumes. You can start it again later.

Terminating an EC2 instance permanently deletes the instance. The root EBS volume is deleted by default, and it cannot be restarted.

Остановка EC2-инстанса выключает его, но сохраняет сам инстанс и подключённые EBS-тома. Его можно запустить снова.

**What is the difference between RDS Read Replicas and Multi-AZ deployments?**

---

Read Replicas are used to scale read traffic by creating asynchronous copies of the primary database. They improve read performance and can be promoted to standalone instances.

Multi-AZ deployment is for high availability. It creates a synchronous standby replica in another Availability Zone for automatic failover, not for scaling reads.

Read Replicas используются для масштабирования чтения за счёт асинхронных копий основной базы. Они повышают производительность чтения и могут быть повышены до самостоятельной БД.

Multi-AZ предназначен для высокой доступности. Он создаёт синхронную резервную копию в другой зоне доступности для автоматического переключения при сбое, а не для масштабирования чтения.

**What is the AWS Shared Responsibility Model?**

---

The AWS Shared Responsibility Model defines that AWS is responsible for security of the cloud (infrastructure, hardware, networking, data centers), while customers are responsible for security in the cloud (data, applications, IAM, OS configuration, encryption, network settings).

Модель общей ответственности AWS означает, что AWS отвечает за безопасность облака (инфраструктура, оборудование, сеть, дата-центры), а клиент отвечает за безопасность в облаке (данные, приложения, IAM, настройки ОС, шифрование, сетевые настройки).

**How do you implement disaster recovery in AWS?**

---

Disaster recovery in AWS is implemented using strategies based on RPO/RTO requirements:

Backup & Restore — regular backups to S3/Glacier, restore in another region when needed.

Pilot Light — minimal core infrastructure running in a secondary region, scale up during disaster.

Warm Standby — fully functional but scaled-down environment in another region.

Multi-site (Active/Active) — fully active environments in multiple regions with Route 53 failover.

Use cross-region replication, automated backups, AMIs, IaC templates, and regular DR testing.

Восстановление после сбоев в AWS реализуется в зависимости от требований RPO/RTO:

Backup & Restore — регулярные бэкапы в S3/Glacier и восстановление в другом регионе.

Pilot Light — минимальная инфраструктура во втором регионе с масштабированием при аварии.

Warm Standby — полностью рабочая, но уменьшенная копия окружения в другом регионе.

Multi-site (Active/Active) — полностью активные среды в нескольких регионах с переключением через Route 53.

Используются cross-region репликация, автоматические бэкапы, AMI, IaC и регулярное тестирование DR.

**What are the key components of a VPC?**

---

The key components of a VPC are Subnets, Route Tables, Internet Gateway, NAT Gateway, and Security Groups.

Subnets divide the VPC into public and private networks.

Route Tables control traffic routing within the VPC.

Internet Gateway enables internet access, while NAT Gateway allows private subnets to access the internet.

Security Groups and Network ACLs control inbound and outbound traffic.

Основные компоненты VPC — это Subnets, Route Tables, Internet Gateway, NAT Gateway и Security Groups.

Subnets разделяют сеть на публичные и приватные сегменты.

Route Tables управляют маршрутизацией трафика.

Internet Gateway обеспечивает доступ в интернет, а NAT Gateway — выход в интернет из приватных подсетей.

Security Groups и Network ACLs контролируют входящий и исходящий трафик.

**How do subnets work in a VPC?**

---

A subnet is a logical subdivision of a VPC within a specific Availability Zone.

It defines a range of IP addresses for resources deployed inside it.

Subnets can be public (with a route to an Internet Gateway) or private (without direct internet access).

Routing is controlled by associated Route Tables.

Subnet — это логическое разделение VPC внутри конкретной зоны доступности.

Она определяет диапазон IP-адресов для ресурсов.

Подсети бывают публичные (с маршрутом к Internet Gateway) и приватные (без прямого доступа в интернет).

Маршрутизация управляется через Route Tables.

**What is the difference between a NAT Instance and a NAT Gateway?**

---

A NAT Instance is an EC2 instance configured to provide NAT services, while a NAT Gateway is a fully managed AWS service.

NAT Gateway is highly available, scalable, and requires no manual maintenance.

NAT Instance requires manual setup, scaling, patching, and high availability configuration.

NAT Gateway provides better performance and reliability but at a higher cost.

NAT Instance — это EC2-инстанс, настроенный для выполнения NAT, а NAT Gateway — полностью управляемый сервис AWS.

NAT Gateway масштабируется автоматически и не требует администрирования.

NAT Instance требует ручной настройки, масштабирования и обслуживания.

NAT Gateway более надёжен и производителен, но стоит дороже.

**What are VPC Peering connections?**

---

VPC Peering is a networking connection between two VPCs that enables private communication using AWS internal network.

Traffic between peered VPCs does not traverse the public internet.

Peering requires non-overlapping CIDR ranges.

It is non-transitive — if VPC A is peered with B, and B with C, A cannot access C automatically.

VPC Peering — это сетевое соединение между двумя VPC для приватного обмена трафиком через внутреннюю сеть AWS.

Трафик не проходит через публичный интернет.

CIDR-диапазоны не должны пересекаться.

Peering не является транзитивным — соединение не передаётся через третью VPC автоматически.

**How does a VPC route table work?**

---

A VPC route table contains rules (routes) that determine where network traffic is directed.

Each route specifies a destination CIDR block and a target (e.g., Internet Gateway, NAT Gateway, VPC Peering).

Every subnet must be associated with a route table.

Traffic is matched against routes, and the most specific route is applied.

Route table в VPC содержит правила маршрутизации, определяющие, куда направляется сетевой трафик.

Каждое правило включает CIDR-назначение и цель (например, Internet Gateway, NAT Gateway, VPC Peering).

Каждая подсеть связана с одной таблицей маршрутов.

Трафик сопоставляется по принципу наиболее специфичного маршрута.

**What is the purpose of VPC Flow Logs?**

VPC Flow Logs capture information about IP traffic going to and from network interfaces in a VPC.

They record details such as source/destination IP, ports, protocol, and traffic status (ACCEPT or REJECT).

Flow logs are used for security analysis, troubleshooting, and monitoring network activity.

Logs can be stored in CloudWatch Logs or S3.

VPC Flow Logs фиксируют информацию о сетевом IP-трафике внутри VPC.

Они записывают исходный и целевой IP, порты, протокол и статус трафика (ACCEPT или REJECT).

Используются для анализа безопасности, диагностики и мониторинга сети.

Логи можно сохранять в CloudWatch Logs или S3.

**How do security groups and network ACLs differ?**

---

Security Groups are stateful and operate at the instance level.

Network ACLs are stateless and operate at the subnet level.

Security Groups allow rules only, and return traffic is automatically allowed.

Network ACLs support both allow and deny rules, and you must explicitly allow return traffic.

Security Groups — stateful и применяются на уровне инстанса.

Network ACLs — stateless и работают на уровне подсети.

Security Groups содержат только allow-правила, ответный трафик разрешается автоматически.

Network ACLs поддерживают allow и deny, и обратный трафик нужно разрешать явно.

Multi-AZ в RDS повышает доступность за счёт создания синхронной резервной реплики в другой зоне доступности.

При сбое основной базы происходит автоматический failover.

Это обеспечивает высокую доступность без ручного вмешательства.

**What is Amazon Redshift, and what are its key features?**

---

Amazon Redshift is a fully managed data warehouse service designed for large-scale analytics.

It uses columnar storage and massively parallel processing (MPP) for high-performance querying.

Redshift integrates with S3, supports SQL-based queries, and scales from gigabytes to petabytes.

It offers features like data compression, workload management, and Spectrum for querying data directly in S3.

Amazon Redshift — это управляемый сервис хранилища данных для аналитики больших объёмов информации.

Использует колоночное хранение и MPP для высокой производительности запросов.

Интегрируется с S3, поддерживает SQL и масштабируется от гигабайтов до петабайтов.

Поддерживает сжатие данных, управление нагрузкой и Redshift Spectrum для запросов к данным в S3.

**What is ElastiCache, and how does it improve performance?**

---

Amazon ElastiCache is a fully managed in-memory caching service that supports Redis and Memcached.

It improves performance by storing frequently accessed data in memory instead of querying the database each time.

This reduces latency and offloads read traffic from backend databases.

ElastiCache is commonly used for caching, session storage, and real-time applications.

Amazon ElastiCache — это управляемый сервис кэширования в памяти с поддержкой Redis и Memcached.

Он повышает производительность, сохраняя часто используемые данные в памяти вместо обращения к базе данных.

Это снижает задержку и разгружает backend-базы.

Используется для кэширования, хранения сессий и real-time приложений.

**How do you implement DynamoDB Streams?**

---

To implement DynamoDB Streams, first enable Streams on the DynamoDB table and choose what to capture (keys only, new image, old image, or both).

Then create an event source mapping to trigger an AWS Lambda function from the stream.

Lambda processes the stream records for actions like auditing, replication, or sending events to other systems.

Use retries and a DLQ/on-failure destination to handle processing errors reliably.

Чтобы внедрить DynamoDB Streams, сначала включают Streams на таблице и выбирают, что писать в поток (только ключи, new image, old image или оба).

Затем настраивают event source mapping, чтобы поток триггерил AWS Lambda.

Lambda обрабатывает записи для аудита, репликации или отправки событий в другие системы.

Для надёжности используют ретраи и DLQ/on-failure destination для ошибок обработки.

**How do database snapshots work in AWS?**

---

Database snapshots in AWS are point-in-time backups of a database instance.

They are stored in S3 and can be used to restore a new database instance at any time.

Snapshots can be automated (scheduled backups) or manual.

During restore, AWS creates a new instance from the snapshot without affecting the original database.

Снимки базы данных в AWS — это резервные копии состояния БД на определённый момент времени.

Они хранятся в S3 и позволяют восстановить новую базу из сохранённого состояния.

Снимки бывают автоматические (по расписанию) и ручные.

При восстановлении создаётся новый инстанс без влияния на исходную базу.

**How can you monitor database performance in AWS?**

---

You can monitor database performance in AWS using Amazon CloudWatch metrics such as CPU, memory, IOPS, and latency.

Enable Enhanced Monitoring and Performance Insights for deeper visibility into database load and query performance.

Use CloudWatch alarms to detect anomalies and trigger notifications.

Logs and slow query logs can also be analyzed for performance bottlenecks.

Мониторинг производительности базы данных в AWS осуществляется через метрики Amazon CloudWatch: CPU, память, IOPS и задержку.

Для детального анализа включают Enhanced Monitoring и Performance Insights.

CloudWatch Alarms используются для обнаружения аномалий и уведомлений.

Также анализируют логи и slow query logs для выявления узких мест.
